# 品牌沙盘 · AI 开发手册（并行集成模式）

> V1.0 | 2026-06-09  
> **模式备注名：并行集成模式** — **3 个模块席** 并行用 AI 改各自目录，**1 个集成主编** 专责合并、规范、部署；**禁止**四人无边界同时改同一文件。  
> 适用：4 人团队（含集成主编），均不会手写代码，希望 **日历时间更短**、愿承担 **更高合并风险**。  
> 更稳、更简单的方式见：`AI开发手册-串行主编模式.md`

---

## 使用说明（先看 3 分钟）

1. 本模式 **不是** 四人各写各的、无人收拾；**集成主编 = 唯一合并权 + 架构守门人**。  
2. 模块席 **只能** 在划定的 `web/app/<模块>/` 内让 AI 改代码（及该模块 API 文件）。  
3. **`prisma/schema.prisma`、登录、`components/ui/`、部署脚本** — **仅集成主编** 可改。  
4. 集成节奏：**每日合并窗**（建议晚 8 点 1 小时），其余时间模块席只 push 自己的 `feat/*` 分支。  
5. 卡住：**「模块名 + 我在 Px 卡住了」** + 截图；模块席 **不自行 merge 到 main**。

**何时选本模式**

| 选并行集成 | 选串行主编 |
|------------|------------|
| 8 周内必须上线、愿每天合并 | 更看重成功率、可 10 周+ |
| 集成主编能 **每天** 投入 1～2 小时合并 | 只有主编一人能长期握 Cursor |
| 三模块席自律、肯用小工单 | 模块席容易「顺手改全局」 |

## 一、角色与模式说明

### 1.1 四人编制

| 角色 | 人数 | 核心职责 | AI 改代码范围 |
|------|------|----------|---------------|
| **集成主编** | 1 | 合并 PR、改 schema、统一 UI、部署、拍板 API、修集成 bug | 全仓库（**合并时**） |
| **模块席 · 品牌** | 1 | 品牌列表、关键人、档案只读页 | `web/app/brands/`、`web/lib/api/brands*` |
| **模块席 · 拜访** | 1 | 安排、记录、日历、承诺 | `web/app/visits/`、`web/lib/api/visits*` |
| **模块席 · 情报** | 1 | 录入、分级、列表、筛选 | `web/app/intelligence/`、`web/lib/api/intel*` |

> 原「验收 / 数据 / 规则」工作：**集成主编兼任**，或三人模块席 **轮流兼** 本模块验收（每人测自己的页 + 周四集成测全流程）。

### 1.2 与「串行主编模式」的称呼对照

| 串行主编模式 | 并行集成模式 |
|--------------|--------------|
| 主编席（唯一编码） | 集成主编（唯一合并） |
| 验收席 / 数据席 / 规则席 | 并入模块席或集成主编兼 |
| 严格 M0→M10 一人推进 | **分轨 Px** 三轨并行 + **集成关口 Ix** |

## 二、产品范围与技术路线

与串行手册 **§二、§三** 相同：拜访 + 情报 + 简品牌 + 工作台；抄 GANT 骨架；独立京东云主机；200 注册 / 50 并发。

```text

手册：docs/AI开发手册-并行集成模式.md
```

## 三、仓库与分支约定

### 3.1 目录边界

```text
web/
├── prisma/schema.prisma      ← 仅集成主编
├── app/(auth)/               ← 仅集成主编（M0 后锁定）
├── components/ui/            ← 仅集成主编（模块席提 Issue）
├── app/brands/               ← 模块席·品牌
├── app/visits/               ← 模块席·拜访
├── app/intelligence/         ← 模块席·情报
├── app/workbench/            ← 集成主编（M5 集成周）
└── lib/api/
    ├── brands*               ← 品牌席
    ├── visits*               ← 拜访席
    └── intel*                ← 情报席
```

### 3.2 分支策略

| 分支 | 谁用 | 规则 |
|------|------|------|
| `main` | 集成主编 | **仅集成主编** merge |
| `feat/brands` | 品牌席 | 每日合并窗前 push |
| `feat/visits` | 拜访席 | 同上 |
| `feat/intel` | 情报席 | 同上 |
| `feat/integration` | 集成主编 | 合并冲突在此解决，测通再进 main |

**模块席禁止**：`git merge main`、`git push main`、改 `schema.prisma`、改他人目录。

## 五、集成主编每日 SOP

| 时段 | 动作 |
|------|------|
| 上午 | 看三席昨夜 push；**字段申请** 排队（改 schema） |
| 合并窗 | 按优先级 merge 一支 `feat/*` → `feat/integration` → 测 build → main |
| 合并后 | 通知三席「明天请 `git pull main` 再开工」 |
| 阻塞 | 帮模块席用 AI 修 **集成层** bug，不要求模块席懂 merge |

**集成主编对 AI 的合并 Prompt 模板：**

```text
请把 feat/visits 合并进当前分支，只解决冲突。
冲突时保留 visits 目录内业务逻辑，保留 main 上 schema 与 auth。
合并后 npm run build，失败则只修到 build 通过。
不要重构 intelligence 或 brands 目录。
```

## 六、模块席每日 SOP

| 步骤 | 动作 |
|------|------|
| 1 | `git pull origin main`（或集成主编指定基线） |
| 2 | 在自己 `feat/*` 上 **只做一个 Px 小目标** |
| 3 | 用 §附录 A 工单问 AI（带目录限制） |
| 4 | 自测本模块页面 |
| 5 | push 到 `feat/*`，**不 merge main** |
| 6 | 合并窗前发「今日完成 Px.x + 验收截图」 |

**模块席禁止**

- 改 `schema` → 填 **字段申请**（附录 B）  
- 改他人 `app/` 目录  
- 「顺便改全局样式」

## 七、字段申请（模块席 → 集成主编）

模块席需要新表/新字段时 **不写 prisma**，只填：

```text
【申请人】拜访席
【表/模型】Visit
【字段】commitmentStatus enum pending|done
【原因】P3 承诺跟踪
【紧急度】本周 I4 前
```

集成主编在合并窗统一改 `schema` 并 `migrate`，再通知三席 pull。

## 八、报错与冲突 SOP

| 情况 | 谁处理 |
|------|--------|
| 本模块页 bug | 模块席 + AI |
| `npm run build` 失败 | 集成主编 |

## 十二、风险（并行特有）

| 风险 | 对策 |
|------|------|
| 合并窗跳过 | 固定每晚；集成主编缺席则 **当日冻结模块席 push** |
| 三席改同一文件 | 集成主编 revert + 重申目录边界 |
| API 不一致 | `API约定-并行版.md` 变更须集成主编签字 |
| 比串行更慢 | 连续 2 次 Ix 失败 → **整组改串行主编模式** |

## 附录 C · 模式选择速查

```text
成功率优先、团队易失控     → 串行主编模式（1 主编席 + 3 辅助席）
```
