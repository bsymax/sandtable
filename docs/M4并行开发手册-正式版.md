## 一、协作规则

> 与 M2/M3 **相同部分不再重复**；完整铁律见 [M2 §1](M2并行开发手册-正式版.md#一协作规则)、[M3 §1](M3并行开发手册-正式版.md#一协作规则)。

### 1.1 能力边界（M4）

| 能力 | M3（已交付） | **M4 新增** |
|------|-------------|-------------|
| 登录/权限 | `admin`/`bd`/`manager`/`readonly` + `user_brands` 过滤 | **`readonly` 禁写** · 写接口 403 · 角色回归全表 |
| LLM | 五路开关 + 超时降级 | **审计日志** · 日配额/用户配额 · `readonly` 默认不写 cache |
| 数仓 | CSV 管道 + `sync_log`（样例/seed 数据） | **BI 首接真 GMV** · 质量规则 · 失败重跑 · `bi_csv` 来源标记 |
| 档案 | Tab1/Tab2 LLM · 数仓指标展示 | **数据截至 YYYY-Wxx** · 口径文档 · **佳璇牵头 BI 对照** |
| 拜访 | 纪要 LLM · 品牌过滤 | **「经营数据已更新至 Wxx」** · 纪要边界加固 · **培翛牵头联调** |
| 情报 | briefing/Feed LLM · CSV | CSV **行级校验** · briefing 与最新周 GMV 一致 |
| 工程 | 外网 117.72.211.51 | **泰山预发** URL（M4-D）；生产切流 → M5 |

**原则**

1. **关 LLM = M2/M3 规则版**；超限/超时降级 **不算 bug**。  
2. **M4 不建驾驶舱**；不做 50 人推广验收。  
3. **数仓真数**：≥1 品牌最新周 GMV 必须来自 BI（非 seed）；佳璇 **D-J-M4-1** 签字。

### 1.3 技术与环境（延续 M3）

栈、页面名、`?brand=`、`api-base.js`、五 `name_key` — **同 M2 §1.3**。

**M4 新增环境变量（Max `.env`，模块席只读约定）**

| 变量 | 含义 | 默认 |
|------|------|------|
| `LLM_DAILY_CAP` | 全站 LLM 日调用上限 | 建议 500（可调） |
| `LLM_USER_DAILY_CAP` | 单用户日上限 | 建议 50 |
| `LLM_READONLY_ENABLED` | `readonly` 是否允许 LLM | `false` |
| `DW_QUALITY_STRICT` | 数仓质量规则严格模式 | `true`（M4-C 起） |

模块席 **不要** 提交含真实 Key 的 `.env`。

### 1.4 公共表与 API（M4 增量）

**M2/M3 铁律仍有效**（只增列、禁止 DROP 培翛 7 表、周报写 `brand_metrics`）。

**M4 预计新增（Max 主导 · 实施前须字段申请）**

| 表/对象 | Owner | 用途 |
|---------|-------|------|
| `llm_call_log` | Max | LLM 审计（user_id, route, tokens, status, latency_ms） |
| `dw_import_batch.source` | Max | 扩展枚举：`bi_csv` / `dts` |
| `data/dw/bi_mapping.json` | Max | `brand_id` ↔ `name_key` |
| `docs/数仓字段口径-v1.md` | 佳璇 | BI 口径书面化 |

**M4 新增/扩展 API（以 Max 合并为准）**

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/llm/audit` | 最近调用日志（admin） |
| POST | `/api/dw/import/csv` | 支持 `source=bi_csv` 批次 |
| — | 各写接口 | `readonly` → **403** |
