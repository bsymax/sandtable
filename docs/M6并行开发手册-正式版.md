## 一、协作规则

### 1.1 M6 相对 M5 的增量

| 能力 | M5（已有） | **M6 新增** |
|------|------------|-------------|
| 账号 | 示例 57 人 · 随机临时密 | **正式 50 人** · **统一初始密** · 工作群发放 |
| 组织 | 单部门语境（厨小 mock） | **双部门** · 顶栏/侧栏 **建材业务部** |
| 品牌 | 5 品牌（卫浴） | **11 品牌**（KA 5 + 基础建材 6） |
| 拜访 | 在线录入 | **历史拜访 Excel 批量导入** |
| 情报 | CSV 运维 | **单条新闻录入** + **隐藏 mock** |
| 工具包 | 无 | **谈参 + 品牌报告**（**培翛** · peixiao-m6-0630 已合并） |
| 高管 | readonly 已有 | **谈参页** 可给负责人/高管只看 |

**原则**

1. **M6 不大改 M5 已交付的改密/admin 流程**；账号导入扩展「统一密码」选项。  
2. **mock 不删库**：用配置/seed 开关 + 列表 API 默认过滤演示数据。  
3. **基础建材真品牌名**：手册与 baseline 用占位；**佳璇 PATCHLOG 写明替换映射**。  
4. **阻塞先 @模块牵头**，再 @Max。

#### J.1 隔离矩阵（三模块）

| 模块 | 隔离维度 | 说明 |

| **数仓 KPI** | **品牌 × `dept_group`** | 与档案经营看板一致 · **各部门各一套** metrics |

```text
权限元组 (brand_id, dept_group)

  ∩ 用户 dept 映射到的 dept_group
  ∩ brand_dept_bindings 中该品牌已开通的部门

  = 用户拥有任一 (brand_id, *) 绑定即可（共享池）
```

#### J.4 API 约定（实现后）

| API | 行为 |
|-----|------|
| `GET /api/auth/me` | 返回 `dept_group` · `brands[]` 含 `{ name_key, dept_scope }` |
| `GET /api/brands` | 按 **(brand, dept)** 有效权限过滤 |
| `GET /api/brands/profile/{key}` | 自动取 **当前用户 dept_group** 的 profile + **同 scope** metrics |
| `GET/POST /api/visits*` | 读写均带 **dept_scope = 用户组** |
| `GET/POST /api/intel/*` | **仅 brand_id**；响应带 `created_by_dept` 展示 |

**角色**：

| role | 范围 |
|------|------|
| `admin` | 全部 brand × dept |
| `manager` | 本 `dept_group` 下 **全部** binding 品牌 |
| `bd` / `readonly` | 本组 ∩ `user_brands` |
