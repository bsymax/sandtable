## 一、index.html 对设计方案的调整（diff 分析）

在对比"设计方案融合版"与"index.html 原型"后，发现 index 做了以下关键调整：

### 1.1 模块裁剪与合并

设计方案规划了 **M1-M14 共 14 个独立模块**，index.html 实际落地时做了简化：

| 设计方案模块 | index.html 中的处理 | 调整逻辑 |
|-------------|-------------------|---------|
| M7 待办指挥中心 | 合并到 M13 工作台 + 全局驾驶舱，不再独立成页 | 待办是渗透多个模块的横切能力，独立页面反而增加跳转成本 |
| M8 全渠道供给分析 | 合并到 M3 预警中心（供给恶化预警）+ 工作台指标卡 | 供给缺口作为预警类型处理，而非独立分析模块 |
| M10 品牌知识库 | 未独立建页，知识提取能力隐含在档案 AI 解读中 | 知识库是后端能力，前端不需要独立展示页 |
| M11 品牌故事库 | 未独立建页，仅在小熊电器档案中提及"沉淀品牌故事" | Phase 4 深化期再做，早期 MVP 不包含 |
| M14 品牌分级 | 渗透到档案（品牌等级标签）、拜访（频率基线）、预警（敏感度）中 | 分级是配置规则而非独立页面 |

**结论**：index.html 把 14 模块收敛为 **7 个导航页面** + 侧边栏，实际需要后端支撑的仍然是对应设计方案的全部能力，但前端路由减少了。

### 1.2 交互模式的升级

index.html 相比设计方案增加了大量交互细节：

| 交互模式 | 设计方案中的描述 | index.html 的实现 |
|---------|----------------|------------------|
| 跨模块跳转 | 描述了"感知→判断→行动"闭环 | 落地为 `switchModule` 函数，支持带参数跳转（品牌、Tab、预填表单） |
| 品牌 Tab 切换 | "按品牌维度加载档案" | 美的/九阳/苏泊尔/小熊/摩飞 5 个品牌，Tab 栏切换 + AI 解读内容差异化 |
| 博弈推演交互 | 描述了推演场景 | 落地为场景卡片选择 + 策略模拟器表单 + 三情景概率结果 + 历史记录筛选 |
| 拜访闭环 | "触发→手册→纪要→追踪" | 落地为 4 个 Tab（安排/记录/手册/日历），表单带 AI 提醒和自动待办生成 |
| Demo 状态标注 | 无 | toast 通知 + "已生成X条待办"反馈 + 文档标注 演示 字样 |

### 1.3 原型数据规模

index.html 作为可演示原型，承载了以下 Demo 数据：

- **5 个品牌**：美的（S级）、九阳（A级）、苏泊尔（A级）、小熊电器（B级）、摩飞（B级）
- **47 条情报**：P0 3 条、P1 12 条、P2 32 条
- **5 条活跃预警**：数据异动 2、关系温度 1、竞对威胁 1、供给恶化 1
- **4 条今日待办**：1 条超期、3 条正常
- **3 个博弈场景**：采销 vs 品牌方（3场景）、品牌间（3 种博弈类型）
- **7 条推演历史**：已采纳 2、部分采纳 2、待验证 2、未采纳 1
- **4 个拜访日历条目**：覆盖美的 6/10 紧急拜访等

### 1.4 对开发的指导意义

index.html 的调整是**有意识的 MVP 裁剪**——它告诉我们：

1. **不需要一开始就做 14 个页面**，核心闭环只需要 7 个页面 + 跨模块跳转
2. **待办和供给分析是横切服务**，不是独立模块，API 设计时就应该作为共享服务
3. **品牌档案是系统的数据中枢**，所有模块都从档案获取品牌上下文
4. **跨模块参数传递是刚需**，前后端都需要支持 `{brand, tab, context}` 的深度链接
5. **Demo 标注的交互需要真实数据替换**，但 UI 布局和交互流程可复用

## 二、技术架构建议

### 2.1 推荐技术栈

| 层次 | 选型 | 理由 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | index.html 约 2856 行，组件化拆分后更可维护；Chart.js 已有成熟 React wrapper（react-chartjs-2） |
| UI 组件库 | Ant Design 5 | 大量表格/表单/卡片/Tab/侧边栏场景，AntD 企业级覆盖度高 |
| 图表 | Chart.js 4 + react-chartjs-2 | index.html 已用 Chart.js，迁移成本最低 |
| 状态管理 | Zustand 或 React Context | 跨模块跳转需要全局品牌上下文，轻量方案即可 |
| 后端框架 | Node.js (Express/Fastify) 或 Java (Spring Boot) | 取决于团队技术栈；AI 推演/生成等 LLM 调用 Node.js 生态更便利 |
| 数据库 | PostgreSQL | 品牌档案的结构化+半结构化数据，JSONB 支持灵活字段扩展 |
| 缓存 | Redis | 经营指标聚合缓存、预警规则热加载 |
| 任务调度 | Bull (Node) 或 XXL-Job (Java) | 情报采集、预警检测、周报生成的定时任务 |
| AI 服务 | LLM API（内部/外部） | 摘要、归因、推演、生成等场景 |
| 部署 | Docker + Nginx | 容器化部署，前后端分离 |

### 2.2 推荐：后端优先用 Node.js

理由：
- AI 推演/生成/摘要/抽取 天然依赖 LLM 调用，Node.js 异步 IO 模型更适配
- index.html 是纯 JS，前后端统一语言降低全栈开发门槛
- 情报采集（爬虫/API 聚合）Node.js 生态成熟

### 2.3 系统拓扑

```

                           │

          ▼                ▼                 ▼
   ┌──────────┐    ┌──────────────┐   ┌──────────┐
   │ 业务服务  │    │  AI 能力中台  │   │ 数据接入  │
   │ 档案/预警 │    │ 推演/摘要/   │   │ BI数仓   │
   │ 拜访/待办 │    │ 生成/抽取    │   │ 外部情报  │
```

## 四、API 设计原则

### 4.1 RESTful 路由规范

```
GET    /api/v1/brands                    # 品牌列表
GET    /api/v1/brands/:id                # 品牌详情（含档案聚合）
GET    /api/v1/brands/:id/profile        # 品牌全景档案（全量）
GET    /api/v1/brands/:id/metrics        # 经营指标时序
GET    /api/v1/brands/:id/contacts       # 关键人
GET    /api/v1/brands/:id/timeline       # 互动时间线
GET    /api/v1/brands/:id/commitments    # 承诺追踪

GET    /api/v1/alerts                    # 预警列表（支持筛选）
GET    /api/v1/alerts?brand_id=&type=&severity=
POST   /api/v1/alerts/:id/acknowledge    # 确认预警
POST   /api/v1/alerts/:id/resolve       # 关闭预警

GET    /api/v1/intelligence              # 情报流
GET    /api/v1/intelligence/weekly       # 情报周报

POST   /api/v1/game-theory/analyze      # 发起博弈推演
GET    /api/v1/game-theory/history       # 推演历史
POST   /api/v1/game-theory/:id/feedback  # 推演反馈

GET    /api/v1/visits                    # 拜访列表
POST   /api/v1/visits                    # 新建拜访
POST   /api/v1/visits/:id/record        # 拜访后记录
POST   /api/v1/visits/:id/brief         # 生成作战手册

GET    /api/v1/todos                     # 待办列表
POST   /api/v1/todos                     # 创建待办
PATCH  /api/v1/todos/:id                 # 更新待办

GET    /api/v1/cockpit                   # 管理驾驶舱聚合数据
GET    /api/v1/workbench                 # 今日工作台聚合数据
```

### 4.2 关键设计决策

- **档案接口返回全量聚合数据**：一次请求获取品牌档案的六大域，避免前端多次请求拼接
- **待办和预警是独立微服务**：被工作台、档案、驾驶舱等多个页面复用
- **推演接口是异步的**：LLM 推演耗时较长，返回任务 ID 后轮询结果
- **周报/手册生成是异步任务**：走任务队列，生成完成后推送通知

### 5.1 难度最高模块详解

**M4 博弈推演引擎**（⭐⭐⭐⭐，25-35 人日）

这是系统的核心差异化能力，也是技术难度最高的模块：

- **前端**：场景选择 + 策略模拟器 + 三情景结果展示 + 历史筛选 + 反馈标记。index.html 已有完整交互模式
- **后端**：Multi-Agent LLM 编排、品牌知识库 RAG 检索、历史案例 few-shot、概率校准
- **AI 能力**：需要构建品牌方行为模型，从历史数据学习品牌方的谈判模式
- **核心挑战**：
  - 推演准确度取决于数据质量和品牌知识库的丰富程度，冷启动阶段置信度普遍偏低
  - LLM 输出的结构化（需要稳定的 JSON schema）
  - 推演耗时长（可能需要 10-30 秒），需要异步处理

**M0-2 AI 能力中台**（⭐⭐⭐⭐，15-25 人日）

所有 AI 能力的基础设施：

- 智能摘要（拜访纪要 → 结构化摘要）
- 异动归因（GMV 下降 + 竞对动作 + 行业趋势 → 自然语言解释）
- 多角色推演（品牌方决策者 + 采销顾问的 Multi-Agent 对话）
- 文档生成（拜访手册、周报、品牌价值报告的模板化 LLM 生成）
- 情感与意图分析（沟通记录 → 品牌方真实诉求和情绪）
- **核心挑战**：Prompt 工程 × 14 个模块的不同场景、输出稳定性、成本控制

**M0-1 数据接入层**（⭐⭐⭐⭐，15-20 人日）

你有内部数据源，这是好消息，但接入仍有挑战：

- BI 数仓的 GMV/UV/转化/毛利/SKU/库存数据同步
- 外部情报（竞对监测、行业报告、全网商品表）的多源采集与清洗
- 人工录入界面（决策链、拜访纪要补充）
- **核心挑战**：不同数据源的更新频率差异（实时/每小时/每天）、数据质量校验、异常标记

### 5.2 中等难度模块

**M1 品牌全景档案**（⭐⭐⭐，20-30 人日）

数据中枢，所有模块依赖它：

- 18 个子功能，index.html 已经做了非常详细的 UI 设计
- 后端最大挑战是 **档案聚合查询**——一次请求需要关联 6+ 张表
- AI 整体解读需要调用 LLM 生成，有上下文长度管理问题
- 档案健康度评分需要实时计算各域完备度

**M6 智能拜访助手**（⭐⭐⭐，15-22 人日）

- AI 自动提取待办（从纪要文本中 NLP 抽取承诺和行动项）是最高难度子功能
- 作战手册生成依赖 M4 推演引擎 + M1 档案数据
- 拜访效果追踪（7/14/30 天自动对比经营数据）需要定时任务

## 六、分阶段实施路线图

### Phase 1 · 基础底座（Week 1-4，目标：数据跑通 + 档案可视）

**目标**：品牌档案页面可展示真实数据，工作台有基本聚合

```
Week 1-2: 项目初始化 + 数据基座
├── 前端项目脚手架（React + TypeScript + AntD）
├── 后端项目脚手架（Express/Fastify + PostgreSQL + Redis）
├── 数据库建表（品牌、联系人、经营指标核心表）
├── 数据接入：经营数据管道（BI → DB）
└── 品牌主数据导入（5 个试点品牌）

Week 3-4: 品牌全景档案 MVP + 工作台
├── M1 品牌全景档案（经营指标卡片 + 产品线 + 渠道占比）
│   └── 先做数据展示，AI 解读用规则模板替代
├── M13 今日工作台（待办列表 + 预警摘要 + 建议联系表）
│   └── 待办服务雏形（手动创建 + 分配 + 状态流转）
├── M14 品牌分级（S/A/B/C 标签 + 基础配置）
└── 权限基础（采销只看自己负责的品牌）
```

**里程碑**：5 个试点品牌档案展示真实经营数据，工作台可手动创建待办

**难度**：数据接入 ⭐⭐⭐⭐ + 档案 ⭐⭐⭐ + 工作台 ⭐⭐  
**人日**：约 25-35

### Phase 2 · 闭环打造（Week 5-8，目标：预警→待办→拜访闭环跑通）

```
Week 5-6: 预警 + 情报 + 拜访基础
├── M3 预警系统（经营异动检测 + 企微通知）
│   └── 先做规则引擎（阈值触发），AI 归因后续迭代
├── M2 情报流（基础版：内部数据异动 + 人工录入情报）
├── M6 拜访助手（安排 + 记录 + 日历，先不做 AI 抽取）
└── 待办升级机制（逾期提醒 + 抄送上级）

Week 7-8: 工作台完整版 + 档案 AI 解读
├── M13 工作台升级（AI 简报 v1 + 建议拜访推荐）
├── M1 档案 AI 解读（经营数据 → LLM → 自然语言摘要）
├── M12 全局驾驶舱（基础版：品牌健康表 + 待办总览）
└── 拜访→待办自动创建（规则触发，非 AI 抽取）
```

**里程碑**：预警触发 → 待办创建 → 拜访记录 → 档案更新，全链路跑通

**难度**：预警 ⭐⭐⭐ + 情报 ⭐⭐ + 拜访 ⭐⭐⭐ + 驾驶舱 ⭐⭐⭐  
**人日**：约 30-45

### Phase 3 · AI 深度（Week 9-14，目标：博弈推演 + AI 能力全面上线）

```
Week 9-11: AI 能力中台 + 博弈推演
├── M0-2 AI 中台搭建（LLM 接入 + Prompt 管理 + 输出结构化）
├── M4 博弈推演引擎（单品牌推演 + 策略模拟器）
│   └── 先支持 3 个高频场景，基于品牌档案 RAG
├── M6 拜访升级（AI 纪要抽取待办 + 作战手册生成）
└── M2 情报升级（外部情报接入 + AI 跨品牌关联识别）

Week 12-14: 战局视图 + 管理驾驶舱完整版
├── M5 品类战局视图（竞争矩阵 + 雷达图 + 关系图谱）
├── M4 品牌间博弈推演（联动谈判建议）
├── M12 驾驶舱升级（品类格局气泡图 + 人效分析）
└── AI 简报/解读全面上线（档案/周报/驾驶舱/作战手册）
```

**里程碑**：博弈推演可生成有价值的策略建议，采纳率 > 50%

**难度**：AI 中台 ⭐⭐⭐⭐ + 博弈推演 ⭐⭐⭐⭐ + 战局视图 ⭐⭐⭐  
**人日**：约 45-65

### Phase 4 · 深化运营（Week 15+，目标：知识沉淀 + 报告生成 + 持续优化）

```
Week 15-18: 资源争取 + 知识库 + 故事库
├── M9 品牌资源争取工具包（一键提案 + 价值报告 PDF）
├── M10 品牌知识库（结构化 + AI 提取 + 过期提醒）
├── M11 品牌故事库（模板化沉淀 + 效果追踪）

Week 19+: 持续迭代
├── 移动端轻量版

├── 绩效系统打通
└── 品牌方人员变动追踪
```

**难度**：资源 ⭐⭐⭐ + 知识库 ⭐⭐⭐ + 故事库 ⭐⭐  
**人日**：约 30-40

## 八、关键风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 数据源不稳定/质量差 | 中 | 高 | Phase 1 建数据校验层；核心页面（档案/预警/待办）不依赖外部数据 |
| LLM 输出不稳定 | 高 | 中 | 所有 LLM 输出附置信度 + 降级模板；关键路径（待办/预警）用规则兜底 |
| 知识库冷启动推演不准 | 高 | 中 | 先用规则引擎 + 人工标注案例，积累 50+ 案例后再上 AI 推演 |
| 前端组件复杂度失控 | 中 | 低 | index.html 已完整定义布局和交互，严格按原型实现不随意扩展 |
| 采销习惯难改变 | 中 | 高 | 从企微推送切入（被动），而非要求开新系统；工作台降低操作成本 |
