## 【人+AI】§1 · 技术路线：别的栈上云是否类似？

### 1.1 结论（先读这一段）

**轻量单机部署的本质是一样的**，可以记成固定三层：

```text
用户 → 边缘入口（Nginx / Caddy）→ 应用进程（uvicorn / PM2 / jar）
                                      ↓
                                 数据库（本机 MySQL / 或云 RDS）
```

| 层次 | 干什么 | 本文示例 |
| --- | --- | --- |
| 边缘 | 静态文件 + 反代 `/api` | **Nginx :80** |
| 应用 | 跑业务 API | **uvicorn + FastAPI :8000** |
| 数据 | 持久化 | **MySQL 8 本机 :3306** |

因此：**本手册以 FastAPI + MySQL + Nginx 写完整示例**；若你的项目是下面「高度相似」的路线，**流程、踩坑、人机分工 80% 可复用**，只需把脚本里的安装命令和进程名换掉。

### 1.2 与本文的相似度（给选型 / 给 AI Fork）

| 技术路线 | 相似度 | 与本文的差异（AI 改脚本时） |
| --- | --- | --- |
| **FastAPI / Flask / Django + MySQL + Nginx** | ★★★★★ | Django 常用 gunicorn；其余同构 |
| **静态 HTML/JS + 后端 API + MySQL** | ★★★★★ | **即本文场景** |
| **Node Express / Koa + MySQL + Nginx** | ★★★★☆ | `npm ci` + **PM2** 代替 uvicorn；其余同构 |
| **纯静态站点**（无后端 API） | ★★★☆☆ | 只需 Nginx 托管文件；**无** systemd 后端、无 DB 步骤 |
| **Next.js 纯静态导出**（`output: export`） | ★★★☆☆ | `npm build` 产物进 Nginx；无 Node 常驻 |
| **Next.js SSR / NestJS 常驻 Node** | ★★☆☆☆ | Nginx 反代 **Node :3000**；构建链不同；内存占用更高 |
| **Next.js / Prisma + 云 RDS PostgreSQL** | ★★☆☆☆ | DB **不在本机**；改连接串、备份、防火墙；应用侧 `migrate deploy` |
| **Docker Compose 一键起** | ★★★☆☆ | 架构仍是三层，但用 **镜像 + compose** 代替 apt/systemd 逐步装 |
| **Java Spring Boot + MySQL + Nginx** | ★★★★☆ | **jar + systemd** 代替 uvicorn；JVM 内存要更大 |
| **云上自建 LLM / GPU 推理** | ★☆☆☆☆ | **另写手册**（GPU 实例、模型文件、显存） |

> **给零代码负责人的一句话**：只要你的项目是「浏览器 + API + 数据库」，就和本文同一种上云思路；若是 SSR、独立 RDS、GPU，让 AI 另写一版 deploy 脚本，**人机分工（§2）和 Pre-flight（§5）仍然适用**。

### 1.3 【人】适用 / 不适用（5 分钟自检）

**适合跟本文走完首次上云**

- Python FastAPI（或同类）+ MySQL 同机 + 静态前端  
- 注册用户百～数百、峰值并发 **≤50**  
- LLM 走**外部 API**，不在云上跑模型  

**请换方案或让 AI 单独立项**

- Next.js SSR、强依赖 Node 常驻  
- PostgreSQL / RDS 与 App 分机部署  
- 本地 7B+ 模型、等保 / 专有云

## 【AI】§2 · 标准架构（以 FastAPI + MySQL + Nginx 为例）

> 其它相似路线：把 uvicorn 换成 PM2/gunicorn/java -jar，MySQL 换成 RDS 连接，**Nginx 反代结构不变**。

```
用户浏览器
    ↓ HTTP :80（后续可加 :443 HTTPS）
京东云 Ubuntu 24.04 轻量主机
├── Nginx
│   ├── /api/*  → 反代 127.0.0.1:8000
│   ├── /docs   → FastAPI Swagger（可选对外关闭）
│   └── /*      → 静态文件 /opt/<app>/web/
├── systemd → uvicorn（建议 2～4 workers，与 CPU 核数匹配）
│   └── FastAPI 入口 main.py
└── MySQL 8（127.0.0.1:3306，不对公网）
    └── 业务库
```

| 层 | 作用 |
| --- | --- |
| Nginx | 静态 + gzip + 反代；生产勿用 `python -m http.server` |
| uvicorn | 只跑 API；worker 数 ≈ CPU 核数 |
| MySQL 本机 | 中小体量够用；变大再迁 RDS |

**生产前端**：去掉 `http://127.0.0.1:8000`，改为同源 `fetch('/api/...')` 或 `API_BASE=''`。

## 【人+AI】§3 · 人机分工

### 【AI】3.1 AI 应独立完成

- 编写/维护：`pack-and-upload.sh`、远程 `*-main-deploy.sh`、`smoke-test.sh`
- 本地打包（排除 `.env`）、`nc -z` 探测、`curl` 轮询
- 读 `/root/*-deploy.log`、`journalctl` 并改脚本
- 更新 §7 踩坑地图（只加**模式行**）

若 `scp`/`ssh` 需要密码 → **停止**，给【人】一行可复制命令。

### 【人】3.2 你必须亲自做（AI 替不了）

| 动作 | 原因 |
| --- | --- |
| 京东云控制台：购机、**防火墙 80/22**、复制 IP | 无控制台权限 |
| **SSH 密码** 只在终端输入 | 禁止发给 AI、群、文档 |
| 浏览器验收、截图 | 确认真实体验 |
| 是否整包重导 schema（可能清库） | 业务决策 |

### 【人+AI】3.3 卡住时怎么说

```text
我在【京东云部署 Step X/10】卡住了
现象：<一句话>
附件：终端最后 30 行 / 浏览器截图
```

AI 应回复：根因（模式级）→ **一条**修复命令 → 要不要重跑全流程。

### 【人+AI】3.4 安全纪律

1. `.env` 不进 tar；服务器 chmod 600  
2. MySQL 密码持久化 `/root/.<app>-db-credentials`  
3. 凭据勿 commit、勿发群、勿写进本文  
4. 有域名后再 certbot HTTPS

## 【人+AI】§4 · 上线前检查清单（Pre-flight）

### 【人】4.1 京东云控制台（约 5 分钟）

- [ ] 镜像 **Ubuntu 24.04**
- [ ] 复制 **公网 IP**
- [ ] 重置 **root 密码**（自己保存）
- [ ] 防火墙：**TCP 22、80**（预留 443）
- [ ] `nc -z <IP> 22` 与 `nc -z <IP> 80` 通

> **80 未放行 = 假失败**，零代码同学最容易卡在这里。

### 【AI】4.2 项目仓库

- [ ] 本地 `/docs` + `GET /api/...` 200
- [ ] `schema.sql` 或 migrate 策略明确
- [ ] 前端生产 API 根可切换
- [ ] `requirements.txt` 完整
- [ ] `deploy/` 三件套齐全（§6）

### 【人+AI】4.3 首次 vs 更新

| 场景 | 做法 | 风险 |
| --- | --- | --- |
| 首次上云 | 整包 `pack-and-upload` | 低 |
| 小改 | scp + restart | 低 |
| 改 schema 且保留数据 | 先 `mysqldump` | 中 |
| 整包重导含 DROP | ⚠️ 可能清库 | 高 |

## 【人】§5 · 套餐怎么选（不用懂技术）

**瓶颈顺序**：带宽 > 内存 > CPU > 磁盘。

| 目标 | 建议 |
| --- | --- |
| ≤10 人内测 | 2C 4G 5Mbps |
| 200 人 / 50 并发试点 | **4C 16G 8Mbps** |
| 50 人同时刷新整页 | 4C 16G **12Mbps+** |
| 外部 LLM API | **不必**为 LLM 买 32G |
| 云上跑大模型 | 不在本文范围 |

LLM 走外部 API 时，服务器只多 HTTP 转发，**套餐按 CRUD 选即可**。

## 【AI】§6 · 部署脚本三件套（Fork 模板）

| 脚本 | 执行位置 | 必须能力 |
| --- | --- | --- |
| `pack-and-upload.sh` | 本机 | 打包 → scp → **nohup** 远程 → curl 轮询 |
| `*-main-deploy.sh` | 云 root | 8 步（§6.1）；可重复；写 deploy.log |
| `smoke-test.sh` | 本机 | HTTP 200 + 关键字 |

### 6.1 远程 8 步（顺序固定）

| Step | 内容 | 防坑 |
| --- | --- | --- |
| 1 | apt 依赖 | **已装则跳过**（防 ssh 断） |
| 2 | MySQL | 密码**持久化** |
| 3 | 解压 `/opt/<app>/` | **chmod web**（防 403） |
| 4 | 前端 API 根 | 优先单文件配置，慎用 sed |
| 5 | venv + pip | 锁 Python 版本 |
| 6 | schema / migrate | 生产慎用 DROP |
| 7 | systemd + Nginx | workers 2～4 |
| 8 | 自检 | curl localhost 健康检查 |

### 6.2 Mac 打包

```bash
export COPYFILE_DISABLE=1
xattr -cr <bundle目录>
```

### 6.3 必须 nohup

```bash
nohup bash /root/<app>-main-deploy.sh /root/<app>-deploy.tar.gz \
  > /root/<app>-deploy.log 2>&1 &
```

## 【AI】§8 · 踩坑地图（部署前必读）

| # | 现象 | 根因 | 修复 | 预防 |
| --- | --- | --- | --- | --- |
| P1 | 外网超时 | 防火墙未开 80 | 控制台加规则 | §4.1 |
| P2 | Permission denied | AI 不能代输密码 | **人本机**跑 deploy | 一行命令 |
| P3 | Connection closed | apt 重启 ssh | nohup + 跳过 apt | §6.3 |
| P4 | Nginx 默认页 | 部署中断 | 重跑 + 看 log | nohup |
| P5 | 静态页 403 | 文件 600 | chmod web | Step 3 |
| P6 | 502 | uvicorn 挂 | journalctl | systemd |
| P7 | MySQL 1045 | 重复部署丢密码 | credentials 文件 | Step 2 |
| P8 | 仍访问 127.0.0.1:8000 | 未切生产 API | api-base.js | Step 4 |
| P9 | sed 报错 | 分隔符冲突 | 改写法或不用 sed | 配置注入 |
| P10 | tar xattr | Mac 扩展属性 | 忽略 | COPYFILE_DISABLE |
| P11 | 数据没了 | 重导 DROP | 先 dump | §4.3 |
| P12 | 冒烟 FAIL | grep 过严 | 放宽断言 | 同步页面 |

## 【人+AI】§9 · 故障排查（先看树，再叫 AI）

```text
外网打不开？
├─ nc 80 不通 → 【人】查防火墙（P1）
├─ Nginx 默认页 → 【AI】tail deploy.log，重跑（P4）
└─ 502 → 【AI】systemctl status <app>（P6）

页面 403？ → 【AI】chmod web（P5）

有页面无数据？
├─ API 非 200 → 【AI】后端日志
└─ API 200 → 【人】硬刷新；查 API 根（P8）
```

【AI】SSH 常用：

```bash
systemctl status <app> nginx mysql --no-pager
journalctl -u <app> -n 50 --no-pager
tail -50 /root/<app>-deploy.log
curl -s localhost/api/health | head
```

## 【人+AI】§10 · 日常更新与回滚

| 变更 | 做法 |
| --- | --- |
| 几个文件 | scp + restart |
| 大版本 | pack-and-upload（**先备份 DB**） |
| 前端 hotfix | scp 或 snapshot rollback |

```bash
mysqldump -u <user> -p <db> > /root/backup_$(date +%F).sql
systemctl restart <app>
```

上线纪律：快照 → 冒烟 → tag（**真实 IP 写私有登记表，附录 A**）。
